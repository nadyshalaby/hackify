**Q2 — Phase 2.5 spec-review audits the work-doc. YOLO has no work-doc. What do the 3 reviewers audit instead?**

## Approach

**YOLO design.** YOLO is full hackify with two gates auto-passed: Phase 2 (Plan sign-off) and Phase 6 (4-options finish menu). Everything else stays — Phase 1 clarifying wizard, Phase 1 exploration step, Phase 2.5 parallel spec-review, Phase 3 parallel implementation waves, Phase 4 verify-with-fresh-evidence, Phase 5 parallel multi-reviewer with auto-fix of all severity tiers (including Critical, per Q1).

**No work-doc.** YOLO never writes to `<project>/docs/work/`. The plan exists only as an assistant message in chat — Original Ask, Q&A, AC list, Sprint Backlog all rendered inline. Phase 2.5 reviewers and Phase 5 plan-consistency reviewer audit that message text. Trade-offs accepted: no pause/resume across sessions, no archived retrospective, reviewers grep chat text instead of structured markdown.

**Phase 6 default.** Auto-pick Option 1 (commit to current branch locally, no push). User inspects with `git log -1` / `git diff HEAD~1` and pushes when ready.

**Quick mode exploration.** Single sentence appended (bolded) to the Phase 1 row in the "Kept phases" table. No new section, no new structure — smallest surgical edit per Q4.

**Why patch and not minor.** Workflow shape additions in YOLO are additive only; quick mode change is one-sentence. No existing user-facing contract is removed or broken. v0.2.3 → v0.2.4 is a patch-level bump.

# Hackify YOLO — Full Discipline, Zero Waiting

Sibling to full hackify. Same workflow phases, zero gates that wait on you.
No work-doc on disk — the plan lives in chat. The user opted into "do it
autonomously"; YOLO does it.

## Workflow shape

Phase 1 (clarify + exploration + wizard if ambiguous)
  → Phase 2 (in-chat plan block — NO doc, NO gate, immediate proceed)
  → Phase 2.5 (3 parallel reviewers audit the in-chat plan)
  → Phase 3 (parallel implementation waves)
  → Phase 3b (debug-when-stuck, if needed)
  → Phase 4 (verify with fresh evidence)
  → Phase 5 (3 parallel reviewers; auto-fix Critical AND Important in-place; log Minor)
  → Phase 6 (auto-pick Option 1: commit to current branch locally, no push; print summary)

## Kept phases — same as full hackify

(table — Phase 1 / 2.5 / 3 / 4 / 5 unchanged)

## What's different from full hackify

| Aspect              | Full hackify              | YOLO                       |
|---------------------|---------------------------|----------------------------|
| Work-doc on disk    | Yes, docs/work/<slug>.md  | NO — in-chat plan only    |
| Phase 2 gate        | Hard gate — waits for go  | NO gate — immediate         |
| Phase 5 Critical    | Surface, ask user         | Auto-fix in-place, no surface  |
| Phase 5 Important   | Auto-fix in-place         | Auto-fix in-place (same)       |
| Phase 5 Minor       | Log to Retrospective      | Log to chat (no doc exists)    |
| Phase 6 menu        | User picks 1/2/3/4        | Auto-picks Option 1 (commit to current branch locally, no push) |
| Pause / resume      | Yes — doc holds state     | NO — close chat = lost      |
| Reviewer audit      | Work-doc Sprint Backlog   | In-chat plan block          |

## When NOT to use YOLO

| Shape                                       | Why                                                   |
|---------------------------------------------|-------------------------------------------------------|
| Multi-day work                              | No work-doc → no resume                              |
| You want to review the plan before code     | YOLO doesn't show you the plan before implementing   |
| Auth / crypto / migration / secret work     | Auto-fix Critical is risky on sensitive surfaces     |
| Cross-team review needs                     | A 4-options finish menu pause is the natural anchor   |
| You can't list expected files up-front      | Same as quick — too underspecified                    |

## Anti-rationalizations

| Thought                                          | Reality                                              |
|--------------------------------------------------|------------------------------------------------------|
| "The user said yolo, skip Phase 1 wizard"        | YOLO skips GATES, not CLARIFY. Run the wizard if    |
|                                                  | the ask is ambiguous; misread asks cost more than   |
|                                                  | a wizard call.                                       |
| "Phase 2.5 has no doc, skip it"                  | The in-chat plan IS the doc. Dispatch the 3         |
|                                                  | reviewers against the assistant message text.        |
| "Critical finding came back, ask the user"       | Q1 locked: auto-fix Critical AND Important; log     |
|                                                  | Minor. The user gets `git diff HEAD~1` after commit  |
|                                                  | — that is the inspection point.                      |
| "Push the commit too — they'll want it on remote"| No. Phase 6 default is local commit only. Push      |
|                                                  | is user-initiated (`git push` themselves).           |

## One-line summary

Full hackify pipeline, no gates that wait on you, no work-doc on disk —
clarify-with-exploration → in-chat plan → spec-review → parallel impl →
verify → multi-reviewer (auto-fix all severities) → commit locally.

## Retrospective

- **What surprised — sync-runtimes.sh is hardcoded.** The MIRROR_SOURCES array is a manual manifest, not a glob. The Phase 2.5 dependency reviewer caught this BEFORE Phase 3 dispatched, saving a guaranteed Phase 4 failure. Future work-docs that add a new skill MUST explicitly include a task to amend `MIRROR_SOURCES`.
- **What surprised — Phase 2.5 catches consistency bugs nothing else would.** The 3 Critical findings (auto-pass token mismatch, commit-locally token mismatch, AC3-vs-anti-rats contradiction) were all in the work-doc itself, not in the eventual diff. Without Phase 2.5, T1's agent would have written a body that fails `[34]`, and we'd have caught it at Phase 4 — losing a wave of work.
- **What surprised — token-match safety.** Phase 5 Security reviewer pointed out `grep -q --` interprets the token as a basic-regex pattern; tokens with `.`/`*`/`[` could false-match. Switched to `grep -qF --` for fixed-string mode. This pre-existing footgun is now closed for `check_token_present`; the sibling `check_no_token` already uses a different (positive-list) idiom and isn't affected.
- **Pre-existing over-cap follow-up.** `scripts/validate-dod.sh` was 676 lines pre-T8 (already over the 500 LOC cap). Post-T8 it's 723 lines. Phase 5 Quality reviewer flagged this; the right fix is splitting by responsibility (e.g., `scripts/validate-dod.d/skills.sh`, `runtimes.sh`) but that is a refactor task, not v0.2.4 scope. Logged for next sprint.
